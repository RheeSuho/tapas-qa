## 1. 배경과 문제 인식

### 수동 테스트만으로는 한계가 있었다

Tapas PC Web의 스모크 TC는 총 213개입니다. 이를 매번 수동으로 실행할 때 다음과 같은 문제가 있었습니다.

| 문제 | 구체적 상황 |
|------|------------|
| **반복 공수** | 배포 때마다 동일한 케이스를 사람이 직접 돌려야 함 |
| **커버리지 한계** | 시간 부족으로 일부 TC만 실행하거나 건너뛰는 경우 발생 |
| **재현 어려움** | 버그 발생 시 어떤 단계에서 깨졌는지 다시 재현해야 함 |
| **야간/주말 공백** | 오후에 배포된 코드가 다음날 오전에야 테스트됨 |
| **회귀 버그 감지 지연** | 이전에 통과했던 기능이 다시 깨졌는지 즉시 알기 어려움 |

### 자동화가 필요하다고 판단한 이유

- 동일한 동작을 반복하는 테스트는 **사람보다 기계가 더 정확하고 빠름**
- QA 인력을 **단순 반복보다 탐색적 테스트와 엣지케이스 발굴에 집중**시킬 수 있음
- 코드 변경이 생겼을 때 **회귀 버그를 배포 전에 잡을 수 있음**

---

## 2. 목표

> **"배포가 완료되면 QA가 버튼 하나로 전체 테스트를 실행하고, 결과를 바로 받는 구조를 만든다."**

구체적으로는:

- Google Sheets에 있는 스모크 TC 213개를 **자동화 코드로 변환**
- 개발팀 배포 완료 알림 → QA가 GitHub Actions **[Run workflow] 클릭 → 테스트 자동 실행 → Slack 알림**
- 실패 시 **스크린샷, 영상, 트레이스**로 즉시 원인 파악 가능
- 결과는 누구나 접근할 수 있는 **대시보드(Allure)**에 자동 게시

## 3. 성과 요약

### 핵심 수치

| 항목 | 수치 |
|------|------|
| **자동화된 TC** | 213개 작성 / 185개 자동 실행 (87%) |
| **1회 전체 실행 시간** | 약 20분 (병렬 실행, headless) |
| **수동 대비 속도** | 약 25배 이상 빠름 ¹ |
| **일일 회귀 실행** | 매 평일 오전 9시 자동 실행 (인력 투입 0) |
| **버그 발견 시 증거** | 스크린샷 + 영상 + 트레이스 자동 첨부 |
| **False Fail** | 자동 재시도로 네트워크 오류 등 일시적 실패 제거 |

> ¹ 213개 TC 수동 실행 시 TC당 평균 2~3분 소요 → 약 7~10시간 vs 자동화 20분

### 공수 절감 추정

| 구분 | 수동 | 자동화 후 | 절감 |
|------|------|----------|------|
| 1회 전체 리그레션 실행 | 약 7시간 | 20분 실행 + 5분 결과 확인 | ~6시간 35분 |
| 월 22일 매일 실행 기준 | 약 154시간 | 약 9시간 | 약 **145시간/월** |

> 실제 매일 전체 스모크를 수동으로 돌리지는 않더라도, 배포 때마다 검증이 필요한 상황에서는 위 수치만큼의 차이가 발생합니다.

### 커버리지

- 기존 Google Sheets 스모크 TC **213개 전체를 코드로 변환 완료**
- 자동화 불가 케이스(탈퇴, 실제 결제 등) 27개는 이유를 명시하고 명시적 제외
- 자동 실행 TC: **186개 (87%)**, 모두 통과 ✅

### 안정성

- 네트워크 지연, 배너 로딩 타이밍 등 일시적 실패 → **자동 재시도 (CI 2회)**
- 실패 시 스크린샷·영상·트레이스가 자동 첨부되어 **재현 없이 원인 파악 가능**
- 히스토리가 누적되어 **언제부터 깨졌는지 추세 확인 가능**

## 4. 어떻게 동작하는가

### 전체 흐름

```
개발팀 배포 완료 알림
          ↓
  QA가 GitHub Actions에서 수동 실행
  (또는 평일 매일 오전 9시 자동 실행)
  ┌──────────────────────────┐
  │  Smoke 테스트 (~1분)     │  ← 배포 직후 핵심 TC만 빠르게
  │  or                      │
  │  Regression 테스트 (~20분)│  ← 평일 매일 오전 9시 전체
  └──────────────────────────┘
          ↓
  Playwright가 실제 Chromium 브라우저를 열어
  tapas.io를 자동으로 조작 + 검증
          ↓
  Allure 리포트 자동 생성 → GitHub Pages 배포
          ↓
  Slack 알림 수신
  ┌──────────────────────────────┐
  │ ✅ Regression Passed          │
  │ 통과 186 / 실패 0 / 20분 │
  │ [📊 대시보드 보기]            │
  └──────────────────────────────┘
          ↓
  실패가 있을 경우:
  Allure 대시보드에서
  스크린샷 + 영상 + 트레이스 확인
```

### 실제로 테스트가 하는 일

자동화 테스트는 **실제 브라우저(Chrome)를 열어서 사람처럼 동작**합니다.

예시 — TPS-042 Comics 카테고리 진입 테스트:

```
1. https://tapas.io 접속
2. 상단 네비게이션에서 "Comics" 클릭
3. Comics 홈 URL로 이동했는지 확인 (/menu/2)
4. 첫 번째 서브탭이 표시되는지 확인
5. 작품 목록이 노출되는지 확인
→ 모두 통과하면 ✅, 하나라도 실패하면 ❌ + 스크린샷 저장
```

## 6. CI/CD 파이프라인

> CI/CD란 코드 변경이 생겼을 때 사람이 개입하지 않아도 자동으로 빌드·테스트·배포가 이루어지는 자동화 파이프라인입니다.

### Smoke 테스트 (배포 후 수동 실행)

**목적:** 개발팀이 배포 완료를 알려왔을 때, 핵심 기능이 살아있는지 빠르게 확인

```
개발팀 "배포 완료" 알림
        ↓
QA가 GitHub Actions에서 [Run workflow] 버튼 클릭
        ↓
GitHub Actions 실행 (smoke.yml)
  - 실제 Chrome 브라우저 설치
  - 저장된 로그인 세션 복원
  - 핵심 @smoke TC만 선별 실행 (~1분)
        ↓
Allure 리포트 자동 배포
  https://rheesuho.github.io/tapas-qa
        ↓
Slack 알림
  ✅ Smoke Passed — 30개 passed (1분 02초)  [대시보드 보기]
```

### Regression 테스트 (평일 매일 오전 9시 자동 실행 / 수동 실행)

**목적:** 이전 배포까지 누적된 변경사항이 기존 기능을 깨트리지 않았는지 전체 확인

```
매일 09:00 KST (평일) 자동 실행
또는 QA가 GitHub Actions에서 수동 실행
        ↓
GitHub Actions 실행 (regression.yml)
  - 전체 TC 186개 병렬 실행
  - 약 20분 소요
        ↓
Allure 리포트 갱신 + 히스토리 누적
        ↓
Slack 알림
  ✅ Regression Passed — 186개 passed (20분)  [대시보드 보기]
```

### 두 테스트 비교

| 구분 | Smoke | Regression |
|------|-------|-----------|
| 실행 시점 | 배포 후 QA 수동 실행 | 매일 오전 9시 자동 / 수동 |
| 실행 TC 수 | 핵심 30개 내외 | 전체 186개 |
| 소요 시간 | ~4분 | ~20분 |
| 목적 | 배포 직후 핵심 기능 빠른 검증 | 전체 회귀 검증 |

### Allure 대시보드 (https://rheesuho.github.io/tapas-qa)

- 매 CI 실행마다 **자동으로 업데이트**
- 브라우저에서 누구나 접근 가능 (별도 로그인 불필요)
- 주요 기능:

| 기능 | 설명 |
|------|------|
| **전체 통과/실패 현황** | 실행 결과 한눈에 확인 |
| **TC별 상세 로그** | 각 단계 소요 시간, 결과 확인 |
| **스크린샷** | 실패 직전/직후 화면 캡처 자동 첨부 |
| **영상** | 실패 TC의 전체 실행 과정 녹화본 |
| **트레이스** | 네트워크 요청, 콘솔 오류까지 포함한 상세 기록 |
| **히스토리** | 과거 실행 결과 누적 → 추세 확인 가능 |

## 8. TC 작성 방식 — Gherkin(BDD)

### Gherkin이란?

테스트 케이스를 **자연어(한국어/영어)로 작성하는 형식**입니다.  
개발자가 아닌 QA나 기획자도 읽고 이해할 수 있으며, 이 문서가 곧 자동화 코드로 실행됩니다.

### 작성 예시

```gherkin
Scenario: [TPS-042] 대메뉴 > Comics 카테고리 클릭
  When 대메뉴 > Comics 카테고리 클릭
  Then Comics 홈화면의 첫 번째 서브탭으로 진입된다.
  And 작품 목록이 노출된다
```

```gherkin
Scenario: [TPS-005] 이메일 로그인
  Given 로그인하지 않은 상태다
  When 이메일과 비밀번호로 로그인한다
  Then 홈 화면으로 이동된다
  And 로그인 상태가 유지된다
```

### 구조 설명

| 키워드 | 역할 | Google Sheets 원본 |
|--------|------|-------------------|
| `Given` | 사전 조건 | Pre-Condition |
| `When` | 실행 단계 | Test Procedure |
| `Then` | 기대 결과 | Expected Result |
| `And` | 이어지는 단계/결과 | 줄 바꿈 |

### 기존 Google Sheets TC와의 관계

```
Google Sheets TC (213개)
        ↓  자동 변환 스크립트
.feature 파일 (74개 파일, 213개 시나리오)
        ↓  playwright-bdd 변환
Playwright 테스트 코드
        ↓  실행
실제 브라우저에서 tapas.io 검증
```

- TC 문서 = 자동화 코드 → **이중 관리 불필요**
- 기존 Google Sheets 내용을 최대한 그대로 유지하여 변환

## 9. 자동화의 장단점

### 장점

| 장점 | 설명 |
|------|------|
| **반복 실행 무한대** | 사람이 지치지 않음. 매일 동일한 품질로 실행 |
| **속도** | 186개 TC를 20분 만에 완료 (수동 대비 ~25배) |
| **회귀 버그 조기 발견** | 코드 변경 즉시 테스트 → 배포 전 버그 감지 가능 |
| **재현 가능한 증거** | 실패 시 스크린샷·영상·트레이스 자동 첨부 |
| **24시간 운영** | 야간·주말에도 자동으로 테스트 실행 |
| **히스토리 추적** | 어떤 시점에 무엇이 깨졌는지 기록으로 남음 |
| **Jira 연동** | 실패 TC에서 버튼 한 번으로 버그 티켓 생성 |

### 한계 및 주의사항

| 한계 | 설명 |
|------|------|
| **초기 구축 비용** | 자동화 코드 작성에 시간 투자 필요 |
| **유지보수 필요** | 화면 UI가 바뀌면 코드 수정 필요 |
| **탐색적 테스트 불가** | 정해진 시나리오 외 예상치 못한 버그는 발견 못함 |
| **동적 콘텐츠 한계** | 프로모션 배너 등 운영 상태에 따라 달라지는 항목은 자동화 어려움 |
| **실제 UX 체감 불가** | "보기 불편하다", "레이아웃이 어색하다" 같은 감성적 판단은 수동 필요 |
| **결제·탈퇴 등 파괴적 작업** | 실계정에 영향을 주는 TC는 자동화 범위 밖 |

### 결론: 수동 테스트와 역할 분담

```
자동화 테스트                  수동 테스트
───────────────────────────────────────────
정해진 시나리오 검증    ←→   탐색적 테스트
회귀 버그 감지          ←→   UX/디자인 감성 검증
반복·빠른 실행          ←→   복잡한 비즈니스 로직 판단
매일 자동 실행          ←→   릴리즈 전 최종 확인
```

> 자동화는 수동 테스트를 대체하는 것이 아니라, **반복 검증을 맡아 QA 인력이 더 중요한 테스트에 집중할 수 있게** 해줍니다.

## 10. 핵심 설계 원칙

### 원칙 1. 코드 직접 작성 (녹화 방식 사용 안 함)

Katalon, Playwright Codegen처럼 화면 동작을 녹화해서 자동화하는 방식을 사용하지 않습니다.

**이유:** 화면이 조금만 바뀌어도 녹화된 좌표/위치 정보가 달라져 테스트 전체가 깨집니다.  
**대신:** 버튼의 이름, 역할, 텍스트를 기준으로 요소를 찾아 **UI 변경에 강한** 코드를 작성합니다.

```
❌ 녹화 방식: "화면 좌표 (230, 450)을 클릭"  ← 레이아웃 변경 시 즉시 깨짐
✅ 코드 방식: "Log In이라는 이름의 버튼을 클릭"  ← 위치가 바뀌어도 동작
```

### 원칙 2. 없어도 죽지 않는 구조

프로모션 배너, 팝업처럼 **있을 수도 없을 수도 있는 요소**는 없어도 테스트가 실패하지 않게 처리합니다.  
있으면 정상 검증, 없으면 "해당 없음"으로 조용히 통과.

### 원칙 3. 자동 재시도 (셀프힐링)

네트워크 지연, 이미지 로딩 타이밍 등 **일시적인 이유로 실패한 케이스는 자동으로 재시도**합니다.
- 로컬: 실패 시 1회 재시도
- CI: 실패 시 최대 2회 재시도
- 재시도 후에도 실패 = 진짜 버그로 판단

### 원칙 4. 실패 증거 자동 저장

테스트가 실패하면 그 순간의 **스크린샷, 영상, 트레이스**를 자동으로 저장합니다.  
QA 엔지니어가 직접 재현하지 않아도 Allure 대시보드에서 바로 확인할 수 있습니다.

### 원칙 5. 로그인 세션 재사용

매 TC마다 로그인하면 시간이 오래 걸립니다.  
처음 한 번 로그인 후 세션을 저장해두고, 이후 모든 TC는 **이미 로그인된 상태**로 시작합니다.  
하루 종일 테스트를 돌려도 실제 로그인은 1회만 발생합니다.

## 13. 로컬 실행 가이드

> 내 PC에서 직접 자동화 테스트를 실행해보고 싶을 때 따라하는 가이드입니다.  
> 개발 경험이 없어도 순서대로 따라하면 실행할 수 있습니다.

---

### STEP 0. 사전 준비 — 프로그램 설치 (최초 1회)

아래 두 가지가 설치되어 있어야 합니다. 이미 설치된 경우 건너뛰세요.

#### Node.js 설치

1. [https://nodejs.org](https://nodejs.org) 접속
2. **LTS** 버전 다운로드 클릭 (왼쪽 버튼)
3. 다운로드된 설치 파일 실행 → 기본값으로 Next 클릭하며 설치 완료

설치 확인 (터미널에서):
```
node -v
```
`v20.xx.x` 같은 버전 번호가 나오면 성공.

#### Git 설치

1. [https://git-scm.com/downloads](https://git-scm.com/downloads) 접속
2. 운영체제에 맞는 설치 파일 다운로드 → 기본값으로 설치 완료

설치 확인 (터미널에서):
```
git -v
```
`git version 2.xx.x` 가 나오면 성공.

> **터미널 여는 방법**  
> - Mac: Spotlight(`⌘ + Space`) → "터미널" 검색 → 실행  
> - Windows: 시작 메뉴 → "cmd" 검색 → 명령 프롬프트 실행

---

### STEP 1. 프로젝트 받기 (최초 1회)

터미널을 열고 아래 명령어를 순서대로 입력합니다.

```bash
# 1. 프로젝트를 내 PC로 복사 (Documents 폴더에 저장됨)
cd ~/Documents
git clone https://github.com/RheeSuho/tapas-qa.git

# 2. 프로젝트 폴더로 이동
cd tapas-qa

# 3. 필요한 패키지 설치 (시간이 약간 걸립니다)
npm install

# 4. 브라우저 드라이버 설치
npx playwright install chromium
```

---

### STEP 2. 계정 정보 파일 설정 (최초 1회)

테스트 실행을 위한 Tapas 계정 정보가 필요합니다.

1. **ruben.lee에게 `.env` 파일을 받습니다** (Slack 또는 이메일로 전달)
2. 받은 `.env` 파일을 `tapas-qa` 폴더 안에 넣습니다

> `.env` 파일에는 테스트용 계정의 이메일과 비밀번호가 담겨 있습니다.  
> 외부에 공유하거나 Git에 올리면 안 됩니다.

---

### STEP 3. 로그인 세션 생성 (최초 1회 + 세션 만료 시)

```bash
npm run test:setup
```

실행하면 Tapas에 자동으로 로그인하고 세션을 저장합니다.  
`기존 세션 재사용` 메시지가 나오면 이미 유효한 세션이 있는 것이므로 건너뛰어도 됩니다.

> 세션은 24시간 유지됩니다. 만료 시 동일 명령어를 다시 실행하면 됩니다.

---

### STEP 4. 테스트 실행

```bash
# 전체 실행 (브라우저 화면 없이 빠르게 — 약 20분)
npm run test:bdd

# 브라우저가 보이면서 실행 (동작을 눈으로 확인하고 싶을 때)
npm run test:bdd:headed

# 특정 TC 하나만 실행 (TPS 번호 입력)
npx playwright test --grep "TPS-042" --project=chromium
```

터미널에 아래처럼 출력되면 정상 진행 중입니다.
```
Running 185 tests using 4 workers
[1/185] ✓ [TPS-001] ...
[2/185] ✓ [TPS-002] ...
```

---

### STEP 5. 결과 확인

```bash
# 실행 결과 리포트 열기 (브라우저에서 자동으로 열림)
npm run report
```

#### 결과 상태 의미

| 상태 | 의미 |
|------|------|
| ✅ passed | 정상 통과 |
| ❌ failed | 실제 실패 — 스크린샷/영상 확인 필요 |
| ⏭ skipped | 자동화 불가 항목으로 의도적 제외 |
| 🔄 flaky | 재시도 후 통과 — 네트워크 등 일시적 문제 |

실패한 TC를 클릭하면 **실패 직전 스크린샷**과 **영상**을 바로 확인할 수 있습니다.

---

### 자주 묻는 문제

**Q. `npm: command not found` 오류가 납니다**  
→ Node.js가 설치되지 않았거나 터미널을 새로 열어야 합니다. STEP 0으로 돌아가세요.

**Q. `git: command not found` 오류가 납니다**  
→ Git이 설치되지 않았거나 터미널을 새로 열어야 합니다. STEP 0으로 돌아가세요.

**Q. 로그인이 안 된다는 오류가 납니다**  
→ `.env` 파일이 없거나 위치가 잘못됐을 수 있습니다. `tapas-qa` 폴더 안에 있는지 확인하세요.

**Q. 테스트 중간에 멈추거나 타임아웃이 납니다**  
→ 네트워크 문제일 수 있습니다. 다시 실행하면 대부분 해결됩니다. 반복되면 ruben.lee에게 문의.

**Q. 프로젝트를 업데이트하고 싶습니다 (최신 코드 받기)**
```bash
cd ~/Documents/tapas-qa
git pull
npm install
```
