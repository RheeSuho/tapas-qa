# Tapas QA 자동화 현황

> 발표자: ruben.lee@dktechin.com · 2026년 6월

---

## 왜 만들었나

Smoke TC가 총 **283개**입니다. 배포 때마다 이걸 사람이 직접 돌리면:

| 문제 | 상황 |
|------|------|
| 반복 공수 | TC 1개당 2~3분 → 전체 약 14시간 소요 |
| 커버리지 한계 | 시간이 부족해 일부는 건너뛰게 됨 |
| 야간/주말 공백 | 오후 배포 → 다음날 오전에야 검증 |
| 환경 분기 어려움 | Prod / QA 환경을 번갈아 수동 검증하면 공수 2배 |

→ **반복 검증은 기계에게 맡기고, QA는 탐색적 테스트에 집중**

---

## 어떻게 동작하나

```
개발팀 배포 완료 알림
        ↓
QA가 버튼 하나로 테스트 실행
(또는 평일 매일 오전 9시 자동 실행)
        ↓
실제 브라우저가 자동으로 tapas.io / m.tapas.io 조작 + 검증
        ↓
결과 → Slack 자동 알림 + Allure 대시보드 자동 업데이트
```

### 실행 모드

| 모드 | 실행 시점 | TC 수 | 소요 시간 |
|------|---------|-------|---------|
| **PC Web Smoke** | 배포 후 수동 실행 | 핵심 30개 | ~2분 |
| **PC Web Regression** | 매일 오전 9시 자동 / 수동 | PC Web 전체 238개 | ~17분 |
| **MWeb Smoke** | 배포 후 수동 실행 | 핵심 ~20개 | ~3분 |
| **MWeb Regression** | 수동 실행 | MWeb 전체 264개 | ~6분 |

### 환경별 실행

명령어 하나로 Prod / QA 환경 전환 가능. 세션 파일, 계정도 자동 분리.

```bash
npm run test:smoke           # PC Web Prod Smoke
npm run test:qa:smoke        # PC Web QA Smoke
npm run test:mweb:smoke      # MWeb Prod Smoke
npm run test:mweb:qa:smoke   # MWeb QA Smoke
```

## 기존 TC를 그대로 자동화 코드로

Google Sheets에 작성된 TC를 그대로 자동화 코드로 변환했습니다.  
형식만 바뀌었을 뿐, 내용은 기존 TC와 동일합니다.

**Google Sheets (기존):**

| 항목 | 내용 |
|------|------|
| No | 42 |
| Pre-Condition | 홈 화면에 접속한다 |
| Test Procedure | 대메뉴 > Comics 카테고리 클릭 |
| Expected Result | Comics 홈화면 첫 번째 서브탭으로 진입 |

**자동화 코드 (변환 후):**

```
시나리오: [TPS-042] 대메뉴 > Comics 카테고리 클릭
  Given 홈 화면에 접속한다              ← Pre-Condition
  When  대메뉴 > Comics 카테고리 클릭    ← Test Procedure
  Then  Comics 홈화면의 첫 번째 서브탭으로 진입된다  ← Expected Result
```

- TC 문서가 곧 실행 코드 → **Google Sheets와 이중 관리 불필요**
- 비개발자도 읽고 이해할 수 있는 형식 그대로 유지
- **동일한 TC를 PC Web과 MWeb에서 모두 실행** — 코드 재작성 없이 플랫폼 확장

## 검증 수준 — C수준 Assertion

단순히 "페이지가 뜨는지"가 아니라 **실제 요소가 보이는지**까지 검증합니다.

```
# 약한 검증 (A수준)
Then 페이지가 로드된다          ← 아무 페이지나 떠도 통과

# 강한 검증 (C수준, 현재 적용)
Then Comics 홈화면의 첫 번째 서브탭으로 진입된다
  → URL이 /menu/2/ 패턴인지 확인
  → 작품 목록 a[href*="/series/"] 요소가 화면에 보이는지 확인
```

- 전체 Then 블록(574개) 중 body만 확인하는 약한 검증: **0개**
- 모든 검증 블록이 URL 패턴 또는 실제 DOM 요소 기반

## DOM 기반 클릭 검증

URL을 직접 입력해서 이동하는 방식이 아니라, **실제 버튼/링크를 클릭해서 이동**합니다.

```
# URL 직접 이동 (검증 없음)
page.goto('/menu/2/subtab/7')   ← 링크가 깨져도 통과

# DOM 클릭 (현재 적용)
page.click('a[href^="/menu/2"]')  ← Comics GNB 링크가 실제로 존재하고 클릭 가능한지 검증
```

DOM inspect를 통해 파악한 실제 selector 사용:
- GNB: `a[href^="/menu/2"]` (Comics), `a[href^="/menu/3"]` (Novels) 등 텍스트 링크
- 서브탭: `a[href="/menu/1/subtab/29"]` (icon-only, href로만 식별 가능)
- 뷰어 툴바: `a.toolbar-btn.js-list-btn` 등 Tapas 전용 CSS class

## 결과 확인 방법

### 1. Slack 알림 — 자동 수신

테스트가 끝나면 Slack으로 결과가 자동 전송됩니다.

**통과했을 때:**
```
✅ Tapas QA — Regression 결과

결과           소요 시간      실행 시각
전체 통과      17분 17초      2026. 6. 1. 오전 9:00:00

통과 210  |  실패 0  |  Skip 28

[📊 대시보드 보기]
```

**실패가 있을 때:**
```
❌ Tapas QA — Regression 결과

통과 208  |  실패 2  |  Skip 28

실패 항목
• [TPS-042] 대메뉴 > Comics 카테고리 클릭

[🐛 TPS-042 Jira 등록]  [📊 대시보드 보기]
```

- **[Jira 등록]** 버튼 클릭 → TC 이름·절차·기대결과가 자동으로 채워진 Jira 티켓 생성
- **Skip 사유**도 같은 이유끼리 묶어서 표시

---

### 2. Allure 대시보드 — 상세 확인

**https://rheesuho.github.io/tapas-qa**

- 별도 로그인 없이 누구나 접근 가능
- 실패 TC 클릭 → 스크린샷 + 영상 + 단계별 로그 확인
- 히스토리 누적 → 언제부터 어떤 기능이 깨졌는지 추세 확인 가능

---

### 3. 로컬 대시보드 — QA 내부망용

QA 서버(`qa.tapas.io` / `qa-m.tapas.io`)는 사내 내부망이라 GitHub Actions에서 직접 접근이 안 됩니다.  
로컬 PC에서 `npm run dashboard` 실행 후 `http://localhost:3333` 접속.

- PC Web / MWeb 선택, Prod / QA 환경 선택해서 실행
- 실행 중 로그 실시간 출력
- 완료 후 Allure 리포트 자동 생성 및 히스토리 누적
- 같은 사무실 네트워크 팀원도 접속 가능
