# Tapas PC Web QA 자동화 현황

> 발표자: ruben.lee@dktechin.com · 2026년 5월

---

## 왜 만들었나

스모크 TC가 총 **238개**입니다. 배포 때마다 이걸 사람이 직접 돌리면:

| 문제 | 상황 |
|------|------|
| 반복 공수 | TC 1개당 2~3분 → 전체 약 9시간 소요 |
| 커버리지 한계 | 시간이 부족해 일부는 건너뛰게 됨 |
| 야간/주말 공백 | 오후 배포 → 다음날 오전에야 검증 |
| 재현 어려움 | 버그 발생 지점을 다시 찾아야 함 |

→ **반복 검증은 기계에게 맡기고, QA는 탐색적 테스트에 집중**

---

## 어떻게 동작하나

```
개발팀 배포 완료 알림
        ↓
QA가 버튼 하나로 테스트 실행
(또는 평일 매일 오전 9시 자동 실행)
        ↓
실제 Chrome 브라우저가 자동으로 tapas.io 조작 + 검증
        ↓
결과 → Slack 자동 알림 + Allure 대시보드 자동 업데이트
```

두 가지 실행 모드:

| 모드 | 실행 시점 | TC 수 | 소요 시간 |
|------|---------|-------|---------|
| **Smoke** | 배포 후 QA 수동 실행 | 핵심 30개 | ~2분 |
| **Regression** | 매일 오전 9시 자동 / 수동 | 전체 238개 | ~17분 |

## 현재 성과

| 항목 | 수치 |
|------|------|
| 자동화 TC | **238개** (원본 213 + 신규 25개) |
| 자동 실행 TC | **210개 (88%)** — 모두 통과 ✅ |
| 1회 실행 시간 | **17분** (수동 약 9시간 대비 35배 빠름) |
| 일일 자동 실행 | 평일 매일 오전 9시 (인력 투입 0) |
| 실패 증거 | 스크린샷 + 영상 자동 첨부 |
| 미운영 콘텐츠 | Spotlight 배너 등 → 자동 skip + 사유 집계 |

> 제외된 TC 28개: 실제 결제, 탈퇴 등 자동화가 불가한 케이스. 이유를 코드에 명시해 관리 중.

## 결과 확인 방법

### 1. Slack 알림 — 자동 수신

테스트가 끝나면 Slack으로 결과가 자동 전송됩니다.

**통과했을 때:**
```
✅ Tapas QA — Regression 결과

결과           소요 시간      실행 시각
전체 통과      17분 17초      2026. 5. 14. 오전 10:26:15

통과 210  |  실패 0  |  Skip 28

[📊 대시보드 보기]
```

**실패가 있을 때:**
```
❌ Tapas QA — Regression 결과

통과 208  |  실패 2  |  Skip 28

실패 항목
• [TPS-042] 대메뉴 > Comics 카테고리 클릭

[🐛 TPS-042 Jira 등록]  [📊 대시보드 보기]
```

- **[Jira 등록]** 버튼 클릭 → TC 이름·절차·기대결과가 자동으로 채워진 Jira 티켓 생성 화면으로 이동
- **Skip 사유**도 같은 이유끼리 묶어서 표시 (예: "Comics Spotlight 서브탭 미운영 상태")

---

### 2. Allure 대시보드 — 상세 확인

**https://rheesuho.github.io/tapas-qa**

- 별도 로그인 없이 누구나 접근 가능
- 실패 TC 클릭 → 스크린샷 + 영상 + 단계별 로그 확인
- 히스토리 누적 → 언제부터 어떤 기능이 깨졌는지 추세 확인 가능

---

### 3. 로컬 대시보드 — QA 내부망용

QA 서버(`qa.tapas.io`)는 사내 내부망이라 GitHub Actions에서 직접 접근이 안 됩니다.  
로컬 PC에서 `npm run dashboard` 실행 후 `http://localhost:3333` 접속.

- Prod / QA 환경 선택해서 실행 가능
- 실행 중 로그 실시간 출력
- 완료 후 Allure 리포트 자동 생성 및 히스토리 누적
- 같은 사무실 네트워크 팀원도 접속 가능
