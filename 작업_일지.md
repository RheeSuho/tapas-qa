## 2026-05-13 (15일차) — 신규 TC 추가 및 CI 속도 문제 해결

### 한 일

#### 1. 신규 TC 18개 추가 (TC-0221~TC-0238)
- 02-홈: Completed / Free Access / WUF 서브탭 TC (빅배너, 더보기, 자동슬라이드)
- 05-홈-(Mature): Daily / Popular / New / Completed 서브탭 TC
- 06-홈-(Community): Early Access / Completed 서브탭 TC

#### 2. CI 속도 문제 원인 파악 및 수정
- 원인: `workers: 1` (순차 실행) + `retries: 2` + 신규 TC 실패 → 22분 → 117분 폭증
- 수정: `workers: 2`, `retries: 1` 로 변경
- sharding(2 job 분할) 시도 후 롤백 — 동일 계정 동시 세션 충돌 위험

#### 3. Spotlight 동적 skip 보완
- 기존: Spotlight 탭 링크 없으면 skip
- 문제: 탭은 있지만 콘텐츠(배너)가 없을 때 fail 발생
- 수정: 탭 클릭 후 배너 콘텐츠 없으면 추가 skip 처리
- 적용 대상: Comics / Novels / Mature / Community Spotlight 4개

#### 4. Mature New > Novels 필터 "No results" 처리
- Novels 신작 없을 때 "No results were found." 문구 노출이 기획 의도
- `날짜별 신작 목록이 노출된다` step에 "No results" 문구도 pass로 처리

### 문제 → 해결

| 문제 | 해결 |
|------|------|
| 신규 TC 추가 후 CI 22분 → 117분 폭증 | workers=1→2, retries=2→1 로 수정 |
| Spotlight 탭 있어도 콘텐츠 없으면 fail | 배너 콘텐츠 체크 후 없으면 동적 skip 추가 |
| Mature New Novels "No results" 시 fail | step에 "No results were found." 문구도 정상 처리 추가 |

### 특이사항
- CI는 workers=2, retries=1 단일 job 구조 유지 (sharding은 세션 충돌 위험으로 채택 안 함)
- Spotlight 동적 skip 로직: 탭 없음 → skip / 탭 있고 배너 없음 → skip / 배너 있음 → 검증
- 오늘 regression 결과 분석 중 (run #13)

- **1차 수정 (잘못된 방향)**: href를 읽어 `page.goto()`로 직접 이동 → "When step은 유저 액션을 검증하는 자리인데 goto()로 대체하면 링크가 실제 클릭 가능한지 검증을 안 하게 된다"는 원칙 위반 지적 → 롤백
- **최종 수정**: `page.evaluate()`로 JS 클릭 적용

> **`page.evaluate()`란?**
> Playwright가 브라우저 내부에 JavaScript 코드를 직접 심어 실행하는 기능.
> 평소 Playwright 클릭은 "요소가 화면에 보이는지 → 안정된 상태인지"를 확인한 뒤 클릭하는데, SPA 재렌더링 중에는 이 확인 과정에서 요소가 사라져버리는 문제가 생김.
> `page.evaluate()`는 브라우저 엔진이 직접 요소를 찾아서 클릭하므로 Playwright의 "안정성 대기" 단계를 건너뛰어 detach 이슈를 우회할 수 있음.
> 단, Playwright의 안전장치를 생략하는 만큼 `display:none` 요소나 SPA detach처럼 불가피한 경우에만 사용.

- **변경 파일**: `steps/홈.steps.ts` — `더보기 링크를 클릭한다` step 수정

## 2026-05-30 (23일차) — Mweb flaky 0건 달성

### 한 일

#### 1. 어제 이어서: 남은 실패 19건 → 0건으로 수렴

22일차에 구축한 모바일 자동화 인프라에서 남은 실패 19건을 하나씩 원인 파악 후 수정했다. 수정을 거치면서 최종적으로 두 프로젝트 모두 0 failed를 달성했다.

| 프로젝트 | 통과 | 스킵 | 실패 |
|---------|------|------|------|
| mobile-safari | 43 | 3 | **0** |
| mobile-safari-common | 202 | 36 | **0** |

#### 2. 발견한 문제와 해결 방법

##### web-to-app 다운로드 팝업이 클릭을 가로막는 문제 (뷰어)

모바일 뷰어에서 추천 작품·댓글 버튼을 클릭하면 "앱 다운로드" 팝업이 먼저 떠서 그 아래에 있는 버튼을 누를 수 없었다.

- 해결: `dismissWebToAppPopup()` 헬퍼를 만들어 팝업 닫기 버튼(`button[data-tiara-action-name="webtoapp_close"]`)을 먼저 닫고 나서 원하는 버튼을 클릭하도록 순서를 바꿨다.
- 적용 범위: 뷰어 추천 작품 클릭, 댓글 버튼 클릭, [더보기] 버튼 재클릭 등

##### 댓글 버튼 두 번째 클릭이 안 되는 문제 (뷰어)

댓글 버튼을 처음 누르면 댓글 창이 모달로 열리는데, 이 모달도 "앱 팝업"처럼 다음 클릭을 가로막는 구조였다.

- 해결: `offsetParent !== null` 조건으로 화면에 실제로 보이는 버튼만 찾아 JavaScript로 직접 클릭했다.

##### 숨겨진 "이어보기" 링크가 먼저 잡히는 문제 (작품홈)

`a[href*="/episode/"]`로 첫 번째 회차를 찾을 때, 화면에 안 보이는 "Continue Reading" 링크가 먼저 잡혀서 120초 대기 후 타임아웃이 났다.

- 해결: 먼저 `a.episode-item`(회차 목록)을 보이는 것만 확인하고, 없으면 `a[href*="/episode/"]`로 폴백하는 순서로 바꿨다.

##### Popular 서브탭이 모바일에서 보이지 않는 문제 (작품홈)

PC 웹에서는 Comics 페이지에 Popular 서브탭이 있지만, 모바일에서는 이 탭이 DOM에는 있어도 화면에 표시되지 않는다. DOM에 있다는 이유로 `.click()`을 호출했더니 120초를 기다리다 타임아웃이 났다.

- 해결: 클릭 전에 `isVisible()` 체크를 추가해서, 보이지 않으면 그냥 현재 페이지에서 계속 진행하도록 처리했다.

##### 드롭다운 메뉴 확인 중 navigation race condition (TPS-207)

Profile을 클릭 → Redeem Code 클릭 → 새 페이지로 이동하는 동안, "하위 메뉴 노출된다" 검증 step이 `isVisible()`로 true를 확인한 직후 페이지가 이동해버려 다시 `expect().toBeVisible()`로 재확인할 때 요소가 사라지는 문제.

- 해결: `isVisible()`로 한 번 확인했으면 바로 통과 처리. 같은 것을 `expect()`로 다시 기다리는 재검증 단계를 제거했다.

##### 로그인 화면에서 이메일 입력창이 가려지는 문제 (TPS-005)

모바일 로그인 화면에 쿠키 동의 배너가 이메일 입력창 위에 겹쳐서, 입력창이 보이지 않아 로그인 코드가 아무것도 하지 않고 건너뛰는 문제.

- 해결: 이메일 입력창 대기 전에 쿠키 배너 동의 버튼을 먼저 클릭하도록 순서를 바꿨다. 이후 첫 시도부터 안정적으로 통과.

##### 세션 파일 없을 때 setup이 120초 후 타임아웃으로 실패하는 문제

`auth.unverified.setup.ts`(미인증 계정 세션 저장)가 세션 파일이 없을 때 로그인을 시도하는데, 각 단계 타임아웃 합계가 최대 145초여서 테스트 전체 타임아웃(120초)을 넘겨버렸다. 이 setup이 실패하면 이에 의존하는 테스트 236건이 전부 실행되지 않았다.

- 해결: 대기 시간 중 가장 긴 `waitForURL` 타임아웃을 90초 → 50초로 줄이고, 로그인 링크 클릭에도 5초 타임아웃을 추가해서 합계가 120초 안에 들어오도록 조정했다.

#### 3. 백 버튼 locator 오탐 수정

`[class*="back"]` 셀렉터가 "feedback" 클래스명도 같이 잡아서 엉뚱한 요소를 클릭하는 문제가 있었다. `aria-label*="back"` + `.btn-back` + `a.back/button.back` 등 명확한 셀렉터로 교체하고, 못 찾으면 `page.goBack()`으로 대신 처리하도록 했다.
