## 2단계. TC 변환 — Google Sheets → .feature 파일

### 기존 상황

QA TC는 Google Sheets에 **213개**가 정리되어 있었습니다.

```
No | 1Depth | 2Depth | 3Depth | Pre-Condition | Test Procedure | Expected Result
```

이걸 그대로 자동화 코드로 쓰려면 변환이 필요했습니다.

### 해결 방법

Python 스크립트(`csv_to_feature.py`)를 만들어 **자동 변환**했습니다.

```
Google Sheets 내보내기 (CSV)
        ↓
csv_to_feature.py 실행
        ↓
.feature 파일 74개 생성
(features/01-공통/, 02-홈/, 03-홈-(Comics)/ ... 섹션별 정리)
```

변환 결과 — 기존 TC가 이렇게 바뀌었습니다:

**변환 전 (Google Sheets):**
```
No: 42
1Depth: 홈
2Depth: 대메뉴
Pre-Condition: 홈 화면에 접속한다
Test Procedure: 대메뉴 > Comics 카테고리 클릭
Expected Result: Comics 홈화면 첫 번째 서브탭으로 진입
```

**변환 후 (.feature 파일):**
```gherkin
Scenario: [TPS-042] 대메뉴 > Comics 카테고리 클릭
  Given 홈 화면에 접속한다
  When 대메뉴 > Comics 카테고리 클릭
  Then Comics 홈화면의 첫 번째 서브탭으로 진입된다
```

- TC 문서가 곧 실행 코드 → **이중 관리 불필요**
- 비개발자도 읽고 이해 가능한 형식 유지

## 3단계. 하네스 엔지니어링 — 어떻게 만들 것인가

### 녹화 방식을 쓰지 않기로 한 이유

Katalon, Playwright Codegen 같은 **녹화·재생 방식**을 검토했지만 채택하지 않았습니다.

| 방식 | 문제점 |
|------|--------|
| 녹화 방식 | 화면 좌표를 기록 → UI 조금만 바뀌어도 전체 깨짐 |
| 코드 방식 | 요소 이름/역할로 찾음 → UI 변경에 강함 |

```
❌ 녹화: "좌표 (230, 450) 클릭" → 버튼 위치 바뀌면 즉시 실패
✅ 코드: "Log In 버튼 클릭" → 위치가 바뀌어도 정상 동작
```

### 하네스 엔지니어링이란?

한마디로 **"테스트 코드도 서비스 코드처럼 제대로 만든다"** 는 접근 방식입니다.

쉽게 비유하면, 수동 테스트에서 "체크리스트"를 쓰는 것처럼 자동화 테스트도 **재사용 가능하고 유지보수 가능한 구조**로 만드는 것입니다.

이 프로젝트에서 실제로 적용한 방식 세 가지:

**① 요소를 위치가 아닌 의미로 찾는다**

화면에서 버튼을 찾을 때 "화면 어느 좌표에 있다"가 아니라 **"이름이 Log In인 버튼"** 으로 찾습니다.

```
일반 사람이 테스트할 때:
  "상단 오른쪽에 있는 파란 Log In 버튼을 누른다"
  → 버튼 위치나 색이 바뀌어도 사람은 알아서 찾음

녹화 방식:
  "화면 좌표 (230, 450)을 클릭한다"
  → 버튼 위치가 바뀌면 엉뚱한 곳을 클릭하거나 실패

하네스 방식 (이 프로젝트):
  "이름이 Log In인 버튼을 클릭한다"
  → 위치가 어디 있든 이름으로 찾으니 바뀌어도 동작
```

**② 반복되는 동작은 한 곳에 모아둔다 (Page Object Model)**

GNB(상단 메뉴바) 클릭은 수십 개의 TC에서 반복됩니다.  
매번 같은 코드를 쓰는 대신 `GnbPage`라는 모듈 하나에 모아두고, TC마다 가져다 씁니다.

```
수정이 필요할 때:

일반 방식: GNB 관련 코드가 50개 TC에 흩어져 있음 → 50군데 다 수정
하네스 방식: GnbPage 한 곳만 수정 → 50개 TC 전체 자동 반영
```

**③ 실패하면 증거를 자동으로 남긴다**

테스트가 실패하는 순간 스크린샷·영상·트레이스를 자동으로 저장합니다.  
다시 직접 재현하지 않아도 Allure 대시보드에서 **실패 당시 화면을 그대로 볼 수 있습니다.**

## 4단계. Step 구현 — 가장 많은 시간이 걸린 구간

### Step이란?

`.feature` 파일의 각 줄이 실제로 브라우저에서 어떤 동작을 하는지 코드로 연결해주는 것입니다.

```gherkin
When 대메뉴 > Comics 카테고리 클릭   ← .feature 파일의 한 줄
```
↓
```typescript
When('대메뉴 > Comics 카테고리 클릭', async ({ page }) => {
  await page.getByRole('link', { name: 'Comics' }).click();  ← 실제 브라우저 동작
});
```

### 진행 과정

처음 구현 후 213개 중 **175개만 통과**, 38개 실패.  
원인을 하나씩 분석하고 수정하는 과정을 **4라운드** 반복했습니다.

| 라운드 | 통과 수 | 주요 발견 |
|--------|---------|----------|
| 1라운드 | 175/213 | 요소 없을 때 30초 타임아웃 발생 |
| 2라운드 | 196/213 | 팝업 처리 방식 문제 |
| 3라운드 | 208/213 | 로그인 세션 유지 문제 |
| 4라운드 | **213/213** | 뷰어 전체화면 JS click 패턴 |

### 핵심적으로 발견한 패턴들

**① Graceful 패턴 — 요소가 없어도 죽지 않는 클릭**

### Graceful 패턴이란?

요소를 클릭하기 전에 **"이 요소가 지금 화면에 있는가?"를 먼저 확인**하고, 없으면 에러 없이 넘어가는 방식입니다.

기존 방식은 요소가 없어도 무조건 클릭을 시도했기 때문에 30초를 기다리다 타임아웃으로 실패했습니다.

```
❌ 기존: 요소가 없어도 클릭 시도 → 30초 대기 → 타임아웃 실패
✅ Graceful: 요소가 있는지 먼저 확인 → 있으면 클릭, 없으면 그냥 넘어감
```

### 언제 써도 되고, 언제 쓰면 안 되는가?

여기서 중요한 구분이 있습니다. Graceful 패턴을 아무 데나 쓰면 **"테스트는 통과했지만 실제로 검증은 안 된" 거짓 pass**가 생길 수 있습니다.

| 스텝 종류 | 역할 | Graceful 패턴 |
|-----------|------|--------------|
| `When` (동작) | 버튼 클릭, 페이지 이동 등 행동 수행 | ✅ 써도 됨 — 원래 없을 수 있는 요소라면 |
| `Then` (검증) | 결과가 올바른지 확인 | ❌ 쓰면 안 됨 — 없으면 반드시 실패해야 함 |

예시로 보면:

```
When  쿠키 배너를 닫는다
      → 쿠키 배너는 있을 수도 없을 수도 있는 UI
      → 없으면 그냥 넘어가도 됨 ✅

Then  Comics 홈 화면이 노출된다
      → 여기서 요소가 없으면 진짜 버그
      → 없을 때 넘어가면 거짓 pass ❌
```

즉, Graceful 패턴은 **"원래 상황에 따라 없을 수 있는 요소를 다룰 때"** 쓰고, **검증 스텝에서는 요소가 없으면 실패**로 처리합니다.

**② 로그인 세션 재사용**

매 TC마다 로그인하면 7초씩 소요 → 213개 × 7초 = **25분 낭비**.  
처음 1회 로그인 후 세션 파일로 저장 → 이후 모든 TC는 **이미 로그인된 상태**로 시작.  
결과: 전체 실행 시간 **절반 이하로 단축**.

**③ Tapas 특수 케이스들**

직접 디버깅하면서 발견한 Tapas 고유의 특성들:

| 발견한 특성 | 처리 방법 |
|-----------|----------|
| 로그인 버튼이 GNB·폼 두 군데에 존재 | `.last()`로 폼 안 버튼 선택 |
| 검색창이 버튼이 아니라 input 태그 | `getByPlaceholder` 사용 |
| `<main>` 태그 없음 | `getByRole('main')` 사용 불가 |
| 뷰어 전체화면 버튼이 `display:none` | JavaScript로 직접 클릭 |
| 일부 페이지 이동이 뒤로가기 불가 | `goBack()` 대신 URL 직접 이동 |

## 5단계. 셀프힐링 — 일시적 실패를 스스로 회복

### 문제

자동화 테스트는 네트워크 지연, 이미지 로딩 타이밍 등 **환경적 이유로 가끔 실패**합니다.  
이게 진짜 버그인지 일시적 현상인지 매번 사람이 판단해야 했습니다.

### 해결 — 자동 재시도

```
실패 감지
    ↓
자동으로 같은 TC 재시도 (로컬 1회, CI 2회)
    ↓
재시도 후 통과 → 일시적 현상으로 기록
재시도 후도 실패 → 진짜 버그로 판단 + 알림
```

### 고려했지만 채택하지 않은 방식

"요소를 못 찾으면 다른 요소를 찾아 클릭해보는" 방식(폴백 체인)도 검토했지만 거부했습니다.  
이유: 엉뚱한 버튼을 클릭해도 테스트가 통과될 수 있어 **거짓 통과(False Pass)** 위험이 있었습니다.

```
❌ 채택 안 함: 못 찾으면 다른 것 클릭 → 엉뚱한 동작인데 통과
✅ 채택:      못 찾으면 재시도 → 그래도 실패하면 진짜 버그
```

## 6단계. CI/CD + Allure 대시보드

### GitHub Actions — 테스트 자동 실행

개발팀이 배포를 완료했을 때, QA가 GitHub에서 버튼 하나로 테스트를 실행할 수 있습니다.  
또는 매일 오전 9시에 자동으로 전체 TC를 실행합니다.

실행 방식:
1. GitHub → Actions 탭 → [Run workflow] 클릭
2. 또는 평일 오전 9시 자동 실행
3. 실행이 끝나면 Slack으로 결과 수신

두 가지 모드:

| 모드 | 실행 시점 | TC 수 | 소요 시간 |
|------|---------|-------|---------|
| **Smoke** | 배포 후 QA 수동 실행 | 핵심 30개 | ~1분 |
| **Regression** | 매일 오전 9시 자동 / 수동 | 전체 186개 | ~4분 |

### Allure 대시보드 — 결과 시각화

모든 실행 결과가 웹 대시보드에 자동으로 게시됩니다.

- **URL**: https://rheesuho.github.io/tapas-qa
- 별도 로그인 없이 누구나 접근 가능
- 실패 TC: 스크린샷·영상·트레이스 자동 첨부
- 히스토리 누적 → 언제부터 어떤 기능이 깨졌는지 추세 확인 가능

```
실행 완료
    ↓
Allure 리포트 자동 생성
    ↓
GitHub Pages에 자동 배포
    ↓
누구나 브라우저에서 결과 확인 가능
```

## 7단계. Slack 알림 + Jira 연동

### Slack 알림

테스트가 끝나면 결과가 자동으로 Slack으로 전송됩니다.  
별도로 대시보드를 열어보지 않아도 통과/실패 여부를 즉시 알 수 있습니다.

**통과 시:**
```
✅  Tapas QA — Regression 결과
결과       전체 통과
통과 / 전체  186 / 186
소요 시간   3분 52초
실행 시각   2026. 5. 7. 오전 9:04:31
[📊 대시보드 보기]
```

**실패 시:**
```
❌  Tapas QA — Regression 결과
결과       2개 실패
실패 항목
• [TPS-042] 대메뉴 > Comics 카테고리 클릭
• [TPS-075] 검색 결과 페이지 진입
[🐛 TPS-042 Jira 등록]  [🐛 TPS-075 Jira 등록]
[📊 대시보드 보기]
```

### Jira 연동

실패 TC의 [Jira 등록] 버튼을 클릭하면 Jira 버그 티켓 생성 화면으로 바로 이동합니다.  
아래 내용이 **자동으로 채워진** 상태로 열립니다.

- 버그 제목: TC 이름
- 발생 경로: 테스트 절차 단계
- 기대 결과: TC의 Expected Result
- 확인 환경: tapas.io PC Web + CI 실행 링크

티켓 내용을 처음부터 작성할 필요 없이 **내용만 보완하고 제출**하면 됩니다.

## 8단계. 품질 강화

### 동적 skip 해소 (15개 → 0개)

초기에는 요소를 못 찾으면 조용히 건너뛰는(skip) TC가 15개 있었습니다.  
통과로 보이지만 실제로는 **검증이 이루어지지 않은 상태**였습니다.

원인을 하나씩 분석해서 전부 해소했습니다.

| 원인 | 해결 방법 |
|------|---------|
| 장르 탭 이미지 alt 텍스트가 다름 (Comics vs Novels) | DOM 직접 분석 후 각각 맞는 locator 적용 |
| 섹션 더보기 링크 구조가 예상과 다름 | 실제 HTML 구조 확인 후 locator 교체 |
| 뷰어 더보기 버튼이 같은 이름 버튼과 혼동 | 위치 기반 조건 추가로 정확한 요소 선택 |
| 이미지 지연 로딩으로 요소 못 찾음 | 대기 시간 추가 |

### C수준 Assertion 재작성

초기 구현은 "페이지가 살아있다"는 수준의 약한 검증이었습니다.  
재작성을 통해 **URL 패턴 + 핵심 DOM 요소 존재** 수준으로 강화했습니다.

```
Before (약한 검증):
  Then 페이지가 노출된다
  → body가 존재하면 통과 — 아무 페이지나 통과됨

After (강한 검증):
  Then Comics 홈화면의 첫 번째 서브탭으로 진입된다
  → URL이 /menu/2 패턴인지 + 서브탭 요소가 존재하는지 확인
```

## 작업 방식 — Claude와 페어 프로그래밍

이 프로젝트 전체를 통해 일관되게 유지한 작업 방식이 있습니다.

```
문제 발생
    ↓
스크린샷 / 오류 메시지 Claude에게 전달
    ↓
원인 분석 + 수정 코드 제안
    ↓
적용 후 테스트
    ↓
통과 or 다음 문제로 이동
```

개발 경험이 많지 않은 상태에서 시작했지만, **모르는 것은 바로 물어보고 적용하는** 방식으로 진행했습니다.
